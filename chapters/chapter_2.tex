\chapter{Problem \& Algorithm Design}
\label{chp:algorithm}

This chapter designs the navigation controller at the core of this thesis. Huang et al.~\cite{huang2026lta} showed that a weight-, size-, and power-constrained micro-drone can reach a single light source in open space, using either a bearing read from a photodiode array or a gradient climbed with a single photodiode. This thesis aims to fill a gap in the work of Huang et al.: reaching several modulated beacons in succession while passing obstacles between them, using only on-board light sensing. We first show the existing methods do not solve this problem. The design question is therefore how to obtain obstacle-aware, multi-beacon navigation from light alone, within the constrained budget of the micro-drone platform we chose.

To overcome these limitations, our controller combines the two methods of Huang et al.~\cite{huang2026lta} and adds a spatial memory of the light field. The bearing points toward a beacon the photodiode array can currently see, while the gradient, estimated from a map of the light the drone makes as it flies, still gives a direction when the beacon is hidden. By fusing the two, the drone steers on the bearing while the beacon is in view and falls back on the remembered light field when an obstacle blocks it. Figure~\ref{fig:fused-overview} shows how the controller derives both headings from the same photodiode array and fuses them into a single heading.

This spatial memory that we add rests on a second change to the sensing. The gradient method of Huang et al.~\cite{huang2026lta} reads the light with a single photodiode, which sees the beacon only while it faces the source, so its estimate depends on the drone's heading. We instead sense with a ring of eight photodiodes, so the photodiode array sees the beacon however the drone is turned. The drone can then record a heading-independent light value at every position it visits, which is what makes the light-field map possible.

The remaining sections develop the controller one component at a time: the bearing estimate, the separation of simultaneous beacons by modulation frequency, the light map, the gradient estimator and the choice among candidate estimators, the state machine that sequences a mission, and the heading and speed commands issued to the flight controller.


\section{Why Existing Methods Fall Short}
\label{sec:existing-methods-fall-short}
In the system proposed by Huang et al.~\cite{huang2026lta}, bearing-angle estimation and gradient-based estimation are first developed as single-beacon navigation primitives and evaluated in source-seeking tasks. This thesis examines two conditions that are more complex and closer to real-world applications than the baseline setting. The first is obstacle occlusion, where an object stands between the drone and the beacon and can block the light signal. The second is multi-target navigation, where the drone must visit several beacons in a predefined order. The following paragraphs explain why the original system is insufficient under these conditions, and then present simulations that demonstrate the failure cases this thesis addresses.


The bearing is an instantaneous estimate, formed from the relative magnitudes of the eight photodiode signals at a single instant as explained in Section~\ref{sec:bearing}. Because it requires no integration over the drone's motion, it yields a heading immediately whenever the array receives the beacon's light, whether directly or through a reflection. This immediacy is also its central limitation: the bearing describes only the direction of the light arriving at that instant and retains no record of the surrounding space. When an obstacle blocks the beacon, every photodiode sees only background light, so the readings no longer differ across the ring. Their weighted sum then points in no particular direction, and the bearing gives the drone neither a heading to follow nor any sign that an obstacle is in the way.

The single-photodiode gradient method, the second method of Huang et al., recovers a direction using one sensor instead of an array. Because a single sensor measures only one light value at a time, this approach is slow and fragile. It cannot sense a spatial gradient directly; instead, it infers the gradient from how the light changes between successive positions. As Figure~\ref{fig:single-pd-gradient} illustrates, this forces the drone to keep moving before it can estimate a direction, while also requiring the sensor to remain oriented toward the light. The method therefore depends strongly on the drone's heading, because one fixed photodiode receives the direct beam only when it faces the source.

We examine these failures in simulation before testing on hardware, for two reasons. A simulated light field carries none of the ambient light or electrical noise of a real flight, so an observed failure can be attributed to the navigation method rather than to the photodiode choice limitation. The simulator also runs the same scenario repeatedly, at low cost and with no risk of damaging the aircraft, which is a critical requirement for comparing different navigation methods.

The comparison uses the Webots environment mentioned in Section~\ref{sec:simulation}, shown in Figure~\ref{fig:sim-setup}. A single beacon sits above the floor, a pillar stands between the drone and the beacon, and the drone starts on one side, so the pillar obstructs the direct path to the beacon. Each controller flies this same scene, from the same start position to the same beacon, with identical sensing and flight settings, so the navigation method is the only variable between runs.

The bearing-only controller steers directly along the measured light direction. Since the pillar lies slightly off the direct line to the beacon, the bearing angle direction just points past its edge, and the controller follows that direction with no indication that it leads into the obstacle. The drone therefore collides with the pillar, as Figure~\ref{fig:bearing-fail} shows.

The single-photodiode gradient controller also fails to reach the beacon. As Figure~\ref{fig:gradient-fail} shows, it follows a slow and irregular path and stalls when it reaches the pillar. This failure comes from how the gradient is estimated. The drone observes a local increase in light intensity only after it has already pressed against the pillar and begun moving around it. Once the sensor detects a stronger light signal after partially passing the obstacle, the gradient estimate produces a large corrective pull away from the pillar. Although this response helps the drone avoid continuous contact with the obstacle, it also pulls the drone away from the target direction, preventing it from reaching the beacon.

The two failures show the trade-off between the two approaches. The bearing is fast and accurate but has no memory beyond the current line of sight. The gradient tolerates a blocked line of sight but is slow and must read from accumulated measurements. A new type of controller is therefore developed in this thesis. We need a controller that maintains a short spatial memory. When an obstacle blocks the beacon, it uses the remembered light field to estimate a gradient and steer around the obstacle. Once past the obstacle, it follows the bearing again. The remainder of this chapter develops this controller. The results are reported in Chapter~\ref{chp:obstacle}.

Neither baseline supports more than one beacon, so neither can perform the multi-beacon traversal this thesis targets. We do not demonstrate the multi-beacon case in simulation. As Section~\ref{sec:simulation} explains, the simulator does not ray-trace the light but reads each sensor's intensity directly, and it does not reproduce the on-off modulation. Without that modulation the beacons cannot be told apart, so we design the multi-beacon navigation in Chapter~\ref{chp:multibeacon}.

\section{System Overview}
\label{sec:overview}

The failures of the previous section define two requirements for the controller. First, the drone must move efficiently toward a visible beacon. Second, it must continue toward the beacon when the direct path is blocked. The controller satisfies the two requirements with two separate algorithms, each addressing one requirement while using data from the same photodiode array. The first is the bearing, which gives an immediate heading whenever the array receives the beacon's light. The second is a gradient, estimated from a light map that the drone builds as it flies; because the map records where the light was stronger, the gradient still gives a direction when the beacon is not directly visible.

Neither algorithm is always trustworthy, so the controller must decide which to use at each moment. The bearing is not reliable when its magnitude is low, which happens when the beacon is weak or blocked as explained in Section~\ref{sec:bearing-angle}. The gradient is meaningless until the drone has measured the light over a spread of positions, as described in Section~\ref{sec:gradient-estimation}. A state machine compares the validity of the two algorithms and selects the drone's behaviour: it follows the bearing estimate while its signal is reliable, adds the gradient once the map is dense enough and its estimate has high confidence, and follows the gradient alone when the bearing is invalid.

Each control update follows the same sequence, summarised in Figure~\ref{fig:controller-overview}. The per-sensor light signals $S_n$ of Section~\ref{sec:lightsignal} are first stored into a light map. The direction toward the light is then estimated from that map through gradient estimation. The bearing and the gradient estimate are then merged into a single target heading by the state machine.

The state machine does more than merge the two estimates. It selects the navigation strategy and forward speed for the current beacon, and it determines when the target has been reached and when the controller should advance to the next.

\section{Bearing Angle}
\label{sec:bearing-angle}

Before the controller can rely on the bearing, it must turn the raw bearing angle of Huang et al.~\cite{huang2026lta} into a usable heading. That estimator, explained in Section~\ref{sec:bearing}, returns an angle from a single set of photodiode readings, but it does not smooth the angle between readings, nor does it say whether the angle can be trusted. This section adds the two steps that supply both: a validity gate that decides whether the bearing is reliable, and temporal smoothing that steadies it.

The bearing is only reliable when the photodiode array actually receives the light from a beacon. With no beacon in view, every photodiode sees only ambient light, so the readings carry no directional information: their vector sum has near-zero magnitude and its angle is set by noise. The bearing cannot flag this on its own, so the controller adds a validity gate. It computes a validity score by summing the light-intensity magnitudes measured across the array, and uses the bearing only when this score exceeds a minimum threshold, which rejects headings that are really noise from a weak, distant, or absent beacon. The gradient is gated separately, by its own reliability conditions in Section~\ref{sec:gradient-estimation}.

A bearing that passes this test is still noisy between samples. Reflections, sensor noise, and small changes in attitude make the raw bearing jitter from one frame to the next, and steering on each raw value would make the drone's heading jerk. The controller therefore smooths the bearing with an exponential moving average, which gives steadier flight. The update is
\begin{equation}
\hat{\psi}_B \leftarrow \hat{\psi}_B + (1-\beta)\,\Delta,
\label{eq:bearing-filter}
\end{equation}
where $\Delta$ is an angular difference between the new bearing and the current estimate $\hat{\psi}_B$, where $\beta$ controls the strength of the filter. A larger $\beta$ rejects more noise but responds more slowly to a genuine change in direction.

\section{Light Intensity Map}
\label{sec:light-intensity-map}

Gradient-based navigation needs a record of how the light varies across space, which the bearing alone does not keep. The controller builds this record as a light map: it divides the horizontal plane into a regular grid and stores one light value per cell, indexed by the drone's estimated position in a fixed world frame. The drone does not survey the area first and then navigate. It fills the map in as it flies, writing the current reading into the cell it occupies on every control update, and estimates the gradient from whatever the map holds so far. Figure~\ref{fig:light-map} shows a survey of a flight area. A real flight fills only a small subset of this map and estimates the gradient from that partial coverage.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.72\textwidth]{figures/gradient_analysis/light_map.png}
    \caption{A light-intensity map of the simulation, viewed from above. Each grid cell holds the strongest light signal recorded while the drone occupied it, drawn here as a smooth field. The strength rises toward the beacon, marked by the star, and this is the scalar field whose gradient the controller follows. The grey disc is the pillar, and the dashed circle the clearance the drone has to keep around it.
    Since the light comes from above and the light sensors are placed on the horizontal plane there is a drop in recorded light strength when very close to or below the light source.}
    \label{fig:light-map}
\end{figure}

Each cell stores the strongest of the eight per-sensor signals $S_n$ measured while the drone occupies it, using the per-sensor signal of Section~\ref{sec:lightsignal}. The controller indexes the cell at the drone's estimated position and updates it on every control update. Because the drone moves slowly relative to the sampling rate, it stays within one cell for multiple consecutive samples, and its path can later cross the same cell again, so each cell is measured multiple times rather than once. Rather than overwrite the stored value with each new sample, the controller blends them through an exponential moving average,
\begin{equation}
I_{\text{cell}} \leftarrow \alpha\,I_{\text{cell}} + (1-\alpha)\,I_{\text{new}},
\label{eq:map-blend}
\end{equation}
which averages out the sample-to-sample fluctuation seen when the same cell is measured repeatedly. The factor $\alpha$ weights the stored value against the new one, set to $\alpha = 0.5$ by default. A smaller $\alpha$ tracks new measurements more closely at the cost of more noise, while a larger $\alpha$ gives a smoother but slower-adapting map.

The cell index depends on the drone's own position estimate, which is not exact. Onboard odometry drifts slowly over time, as all dead-reckoning estimators do. As it drifts, measurements are gradually stored in the wrong cells, smearing the recorded intensity across neighbouring cells and blurring the gradient estimated from the map.

Two design choices limit the effect of this drift. First, the map is reset whenever the drone switches to the next beacon, so the error accumulated while approaching one beacon is discarded with its map rather than carried into the next, and drift cannot build up across a whole mission. Second, the map is used only to estimate a heading, not the beacon's absolute position. Even as the position estimate drifts, the direction in which the light rises fastest across nearby cells stays accurate enough to steer by.

Across the visited area the map approximates a scalar field that rises toward the beacon, because the measured signal grows as the drone nears the source. This field is the spatial memory the bearing lacks, and the next section explains how we estimate a direction from it.

\section{Gradient Estimation}
\label{sec:gradient-estimation}

The gradient gives the drone a direction to follow when the bearing is unavailable. It derives a different heading from the bearing: the bearing points along the light arriving now, whereas the gradient points in the direction in which the stored light field increases fastest, which remains defined even when the beacon is hidden. Huang et al.~\cite{huang2026lta} estimated this direction from a single photodiode, by reading how that sensor's signal changed as the drone moved. That approach is slow, because the drone must keep moving to gather readings, and fragile, because a single fixed sensor measures the source only when it faces the light. Both of these problems can be seen in Section~\ref{sec:existing-methods-fall-short}. Our controller instead estimates the gradient from the light map described in Section~\ref{sec:light-intensity-map}. Because the map is built from the full photodiode array, it records a usable value at each visited location regardless of the drone's heading. This removes the dependence on orientation and motion that limits the single-photodiode method.

The controller estimates the gradient from the recorded light map by fitting a plane to it. Around the drone's current position it takes the cells $\{(x_i, y_i, I_i)\}$ and fits $I(x,y) = a x + b y + c$ by weighted least squares, weighting each cell by the inverse square of its distance to the drone so that nearer cells have greater influence on the estimate. The fitted coefficients give the gradient $\mathbf{g} = (a, b)$, and the controller steers along $\mathbf{g}$ when navigating on the gradient.

A gradient estimate, like a bearing, can be unreliable, so the controller needs a measure of its validity. The plane fit supplies one directly. If the nearby cells lie close to the fitted plane, the slope it gives is trustworthy; if they are scattered, it is not. We quantify this agreement with a weighted coefficient of determination, $R^2$, which measures the fraction of the variation in the cell values that the fitted plane explains,
\begin{equation}
R^2 = 1 - \frac{\sum_i w_i\,(I_i - \hat{I}_i)^2}{\sum_i w_i\,(I_i - \bar{I}_w)^2},
\label{eq:gradient-r2}
\end{equation}
where $\hat{I}_i$ is the plane's value at cell $i$, $\bar{I}_w$ is the weighted mean of the cell values, and $w_i$ are the inverse-distance weights. An $R^2$ near one means the cells lie close to a single plane, so the fitted direction is trustworthy. A low $R^2$ means they do not, so the gradient should not be followed.

Three conditions must hold before the controller can trust a gradient estimation result. First, the fitting process must use at least a minimum number of cells, because a plane fitted through only two or three points is usually noisy and can produce a random pointing direction. Second, its $R^2$ must exceed a threshold, which rejects the fitting results where the cells do not lie on a consistent slope, as happens where the field is flat or where the map is still patchy. Last, its magnitude must be large enough to be meaningful, so the drone does not chase a near-flat field. The thresholds, with the controller's other parameters, are listed in Appendix~\ref{app:parameters}. These three conditions define the gradient reliability used by the state machine. While they fail, the drone navigates on the bearing alone. Once they hold, the controller blends the gradient into the heading, or follows it alone when the bearing is lost.

\subsection{Selection of the Gradient Estimator}
\label{sec:gradient-estimator-comparison}

Estimating a gradient from sparse and irregularly spaced map cells is a well-studied problem, but it is not obvious which estimator is best suited to the light maps produced during flight. Different methods make different trade-offs between robustness, accuracy, computational cost, and the ability to produce an estimate when map coverage is limited.

Since the gradient estimator runs continuously on the drone's microcontroller, the chosen method must not only be accurate but also execute within the available computational budget. We therefore compare several candidate estimators before selecting the one used in our experiments. Five candidate methods were evaluated:

\begin{itemize}
    \item Ordinary least-squares plane fitting (OLS) \cite{bjorck1996lsq}.
    \item Weighted least-squares plane fitting (WLS) \cite{strutz2016wls}.
    \item RANSAC plane fitting \cite{fischler1981ransac}.
    \item Local polynomial surface fitting \cite{bjorck1996lsq}.
    \item Finite-difference estimation \cite{leveque2007fdm}.
\end{itemize}

All methods estimate the local gradient from neighbouring map cells but differ in how they treat noise, sparse sampling, and outliers.

\subsubsection{Evaluation Procedure}

The methods were evaluated using a densely sampled reference map generated from the simulated light field. This reference map includes an obstacle creating a shadow. This lets us evaluate the estimators on the task they are actually used for: guiding the drone around obstacles and out of shadows. At each map location a method either produced a gradient estimate that passed the reliability criteria of Section~\ref{sec:gradient-estimation} or returned no estimate. Three criteria were considered:

\begin{enumerate}
    \item Success rate: how often a usable estimate is produced.
    \item Directional accuracy: how closely the estimated direction matches the
        direction towards the light.
    \item Computational cost: the processing required for a single estimate.
\end{enumerate}

Success rate is the most important criterion because the gradient's primary purpose is to remain available whenever the bearing is lost. An estimator that is slightly less accurate but available most of the time is more useful to the controller than a highly accurate estimator that frequently returns no result.

Directional accuracy was measured using the angular error:
\begin{equation}
e_i = \left| \left( \hat{\theta}_i - \theta_i^\star + 180^\circ \right) \bmod 360^\circ - 180^\circ \right|,
\label{eq:angle-error}
\end{equation}
where $\hat{\theta}_i = \operatorname{atan2}(\hat{b}_i, \hat{a}_i)$ is the estimate's heading and $\theta_i^\star = \operatorname{atan2}(y_L - y_i,\, x_L - x_i)$ points from cell $i$ to the source at $(x_L, y_L)$, and the wrap folds the unsigned difference into $[0^\circ, 180^\circ]$. When computing the error we discard cells within ten centimetres of the source, where the field is too steep to give a meaningful reference.

% It is important to note that this straight-to-source reference is itself an approximation when the light signal is blocked by the pillar. Under occlusion, the useful direction is not necessarily the geometric direction toward the light source, but the direction that helps the drone move away from the pillar and recover the light signal.


Directly comparing average angular error using the same number of data points can be misleading, because the methods estimate the gradient in different ways and therefore require different amounts of local map data before they can produce a valid estimate. Some methods return an estimate almost everywhere, whereas others withhold an estimate when the local map is sparse or poorly fitted. As a result, the average error for each method is computed only over the locations where that method produces a valid estimate, rather than over the same set of time steps for all methods. Figure~\ref{fig:grad-own-error} reports this own-coverage error for each method.

While angular accuracy is important for drone navigation, the coverage of a gradient estimator is also critical. We therefore report an additional coverage metric, defined as the estimation success rate. For each estimator, we divide the number of map cells where it produces a valid gradient estimate by the total number of cells in the map. Figure~\ref{fig:grad-success} reports this success rate, the ratio of the map over which each estimator can provide a usable navigation direction.

To provide a fairer comparison of angular accuracy, we also evaluate the estimators on the common set of map cells where all candidate estimators return valid results, which Figure~\ref{fig:grad-common} reports. In this analysis, a cell is included only if every estimator produces a valid gradient estimate at that location. This avoids comparing methods over different parts of the map and isolates the accuracy difference among estimators by evaluating them under the same spatial conditions.


The distinction is important. Consider two methods: one returns estimates at 95\% of locations, while another returns estimates at only 70\%. The lower-coverage method can avoid many of the difficult locations simply by refusing to estimate there. Its average error may therefore appear lower, not because its estimates are inherently better, but because it is evaluated only on easier cases. The common success points comparison removes this effect by ensuring that every method is scored on exactly the same locations.

This behaviour is visible in the results that we will show later. Weighted least squares appears substantially less accurate when evaluated over its own coverage because it continues to provide estimates in difficult regions, including locations close to the source and behind the obstacle. When the comparison is restricted to the common subset, where all methods are evaluated on the same points, the differences between the plane-fitting methods become much smaller. This indicates that most of the apparent accuracy gap is caused by differences in coverage rather than differences in the quality of the estimated direction itself.

We consider the success rate first, as it is the most important criterion. As Figure~\ref{fig:grad-success} shows, the finite-difference estimator is available almost everywhere, with a success rate of 99.7\%, and weighted least squares follows closely at 95.9\%. The polynomial surface and ordinary least squares cover 91.9\% and 81.9\% of the map, while RANSAC covers the least, at 72.3\%. Because the gradient's main purpose is to remain available whenever the bearing is lost, this coverage is the metric that matters most.

The own-coverage error in Figure~\ref{fig:grad-own-error} must be read against these coverage figures. On its own valid cells RANSAC has the lowest mean angular error, at 5.9\textdegree{}, followed by ordinary least squares and the polynomial surface at 13.6\textdegree{} and 13.9\textdegree{}, while weighted least squares and finite differences appear worse, at 29.2\textdegree{} and 36.6\textdegree{}. This ranking is misleading on its own. The high-coverage methods keep producing estimates in the difficult regions close to the source and behind the obstacle, where pointing straight at the source is no longer the useful direction, and those hard cells inflate their average. RANSAC's low error comes partly from declining to estimate the hard cells at all.

When all methods are evaluated on the common set of 284 cells that every method estimates, shown in Figure~\ref{fig:grad-common}, this coverage advantage is removed and the quality of the estimated direction itself is isolated. On this common set the plane-fitting methods fall within a few degrees of one another, at 8.6\textdegree{}, 9.5\textdegree{}, 5.9\textdegree{}, and 6.3\textdegree{} for ordinary least squares, weighted least squares, RANSAC, and the polynomial surface, while finite differences remain noticeably noisier at 15.3\textdegree{}. Most of the apparent accuracy gap seen on each method's own coverage is therefore caused by which cells are estimated rather than by the intrinsic quality of the estimated direction.

\subsubsection{Selected Method}

After our evaluation, we select the weighted least squares method as our final implementation for our controller. Although several methods achieve similar directional accuracy, weighted least squares provides the best balance between the three evaluation criteria. It offers high enough coverage, low computational cost, deterministic execution time, and robustness to variations in measurement density and overall light level.

The computational cost follows from how each method uses the $n$ cells in the local neighbourhood, which a fixed search radius keeps to a few tens of cells. Ordinary and weighted least squares, the polynomial surface, and the finite difference each build a small linear system of fixed size from the $n$ cells and solve it once, so each costs $O(n)$ per estimate. The polynomial surface carries the largest constant factor, as it solves for six coefficients rather than three. RANSAC instead repeats a minimal-sample plane fit over up to fifty random subsets and scores all $n$ cells each time, giving $O(kn)$ for $k$ iterations. This is the highest cost of the five, and its iteration count makes its execution time non-deterministic.

Finite differences provide slightly higher coverage but at the cost of substantially noisier estimates. RANSAC produces accurate estimates where it succeeds but has the lowest coverage and, as its complexity shows, the highest computational cost.

Weighted least squares therefore provides the most suitable compromise for real-time deployment on the drone's microcontroller and is used throughout the experiments in this work.

\section{Multi-Beacon Traversal}
\label{sec:multibeacon-separation}

In the real-world application, when multiple targets are present, the controller must be able to distinguish different beacons. We make the beacons distinguishable by modulating each one at a distinct frequency. Frequency analysis of a single photodiode signal then produces a separate peak for each beacon, allowing the photodiode array to sense multiple beacons at the same time. The traversal remains sequential. The controller selects one beacon as the current target, flies toward it, and selects the next beacon once the target is reached.


In a real-world environment, a photodiode reports a single mixed light-intensity value at each timestamp, where signals from all beacons are combined with steady background illumination. By collecting a sequence of intensity measurements over time, the controller can apply a Fourier transform to decompose the mixed signal into its frequency components. Because each beacon is modulated at a distinct frequency, the transformed signal contains an energy peak at each beacon's modulation frequency, and the height of that peak reflects how strongly the corresponding beacon is observed. Figure~\ref{fig:fft-mock} illustrates this process using generated data for two beacons modulated at 150 and 200\,Hz. Their signals overlap in the time domain but separate into distinct peaks in the frequency domain. The Fourier transform output is then passed to a peak-extraction step, which recovers the signal strength at each target frequency, whose implementation details are described in Chapter~\ref{chp:realworld}.

Two lights are switched on and off as square waves at 150 and 200\,Hz, and one photodiode sees their sum as an ADC reading between 0 and 4095. Each photodiode reading in the array is passed through a Fourier transform to produce its own frequency-domain response. The peak heights at the modulation frequencies differ across the array according to each photodiode's orientation relative to the light beacon. We then use these peak heights, i.e., the frequency magnitudes, as the light-signal inputs defined in Section~\ref{sec:bearing}. The bearing estimator processes these values and produces a bearing angle toward each beacon based on its modulation frequency.


Because real-world environments may contain noisy light sources, such as computer displays or illuminated signs, the modulation frequencies must be chosen carefully. Switching a light fully on and off produces a square wave. Although this modulates the beacon at the desired target frequency, it also introduces odd harmonics~\cite{oppenheim1999dsp}. These odd harmonics can also be seen in Figure~\ref{fig:fft-mock}. The frequency analysis separates the fundamental target frequency from its harmonics because they fall into different frequency bins. A problem arises only when one beacon's fundamental frequency coincides with a harmonic of another beacon. In that case, the harmonic adds energy to the wrong beacon's bin, making the two beacons difficult to distinguish. Therefore, the target modulation frequencies must be selected to avoid harmonic collisions. For two target beacons, we choose frequencies such as 150\,Hz and 200\,Hz.


\section{Controller State Machine}

\label{sec:state-machine}
The bearing and gradient methods each provide a direction estimate, but they do not by themselves decide what behaviour the drone should adopt at a given moment. Two situations must be considered before the drone acts. In the first, the drone may not have a trustworthy estimate, either because it has just started searching for the target beacon and the map is still empty, or because the light is blocked by an obstacle. In the second, both estimates are available, and the controller must decide which estimate to rely on while the drone moves. We resolve both situations with a navigation state machine, shown in Figure~\ref{fig:state-machine}, which consists of five states, namely \emph{searching}, \emph{aligning}, \emph{approaching}, \emph{dwelling}, and \emph{recovering}. The following paragraphs describe each state and its purpose.


In the \emph{searching} state, the drone has no trustworthy bearing or gradient estimate, and its purpose is to search for a beacon. Because the photodiode array already covers the full 360-degree horizontal plane, rotating in place would reveal little additional information. The drone therefore flies a bounded figure-eight search path instead. This pattern forces the drone to explore the surrounding area, increasing the chance that it passes through regions where the beacon signal is stronger while recording light measurements into the map. The drone leaves \emph{searching} as soon as the received signal produces a bearing whose magnitude exceeds the trust threshold. During this state, the drone does not use the gradient or bearing estimates to command its motion; it follows only the predefined search path while monitoring the signal strength.


Once the drone has collected enough light information to infer the target direction, either from the bearing estimate or from the gradient estimate, it transitions into the \emph{aligning} state. In this state, the drone turns toward the target while holding position, allowing it to face the beacon before committing to forward motion. It transitions to \emph{approaching} once the drone's heading falls within the alignment tolerance of the target direction, set to $15^\circ$ in this work. Appendix~\ref{app:parameters} lists this parameter together with the other controller parameters.


In the \emph{approaching} state, the drone flies forward while continuously correcting its heading toward the target. However, odometry drift can gradually shift the drone's heading away from the estimated target direction. If the heading error later exceeds a wider re-alignment threshold of $30^\circ$, the controller returns to the \emph{aligning} state, ensuring that the drone does not continue flying forward while severely misaligned.


The heading used during \emph{aligning} and \emph{approaching} is not always determined by the bearing alone. At the start of the mission, when the map is still sparse, the drone follows the bearing estimate because too few cells have been recorded to support a reliable gradient. Once the gradient satisfies the reliability conditions in Section~\ref{sec:gradient-estimation}, the controller blends the bearing and gradient estimates into a single target heading. In this way, the drone begins using both estimates during normal approach, before the bearing is lost, making the later transition to gradient-only control smoother when the bearing estimate becomes invalid.

The drone must also detect arrival at a beacon. The measured light strength rises as the drone nears the source, so the drone is treated as arrived once this strength exceeds an arrival threshold. This threshold is tuned per flight, with its value listed in Appendix~\ref{app:parameters}. On arrival the drone enters the \emph{dwelling} state, holds position for a fixed dwell time of $1.0$\,s, and then moves to the next beacon. 

The \emph{recovering} state handles the case where the bearing becomes invalid in flight, for example when an obstacle moves between the drone and the beacon. The magnitude of the bearing estimate then drops below the trust threshold, and a controller without memory would have no heading to follow. However, if the gradient estimate meets its reliability conditions, the drone enters \emph{recovering} and continues on the gradient direction alone, climbing the stored light field to move out of the obstacle's shadow. It returns to \emph{approaching} as soon as the bearing is valid again, and falls back to \emph{searching} only if the gradient estimate also becomes unreliable.

\section{Control Output}

The controller's primary output is a single target heading, obtained by merging the bearing direction $\psi_B$ and the gradient direction $\psi_G$. Both are angles, so the controller combines them with a weighted circular mean~\cite{mardia2000directional},
\begin{equation}
\psi = \operatorname{atan2}\!\big(w_B \sin\psi_B + w_G \sin\psi_G,\; w_B \cos\psi_B + w_G \cos\psi_G\big),
\label{eq:fusion}
\end{equation}
A plain numerical average is unsuitable here because angles wrap at $360^\circ$. Averaging $359^\circ$ and $1^\circ$ gives $180^\circ$, the opposite of the correct direction, whereas the circular mean returns $0^\circ$.

The weights $w_B$ and $w_G$ follow the validity of each input, as decided by the state machine. When both are valid, the controller sets $w_B = w_G$, so the heading is their bisector. When only one is valid, its weight is one and the other zero, and the controller follows that input alone. When neither is valid, there is no direction to command, and the drone falls back to the \emph{searching} state rather than holding a fixed heading.

The controller also needs to set a forward speed once it has found a reliable target direction. The drone flies forward only when it is aligned with the target, so its forward speed is set to zero while it turns during the \emph{aligning} state. Holding the speed at zero during large heading corrections prevents the drone from drifting sideways. Once the heading falls within the alignment tolerance, the controller enters the \emph{approaching} state and commands a fixed forward speed toward the target. The heading and forward speed are sent to the platform's flight controller as setpoints, while the altitude is held at a fixed height throughout the experiment. The navigation controller therefore operates only in the horizontal plane. Because it communicates with the platform only through heading and speed commands, the same navigation logic is independent of altitude in our experiments. The same implementation runs in both the Webots simulation and the real Crazyflie platform.
