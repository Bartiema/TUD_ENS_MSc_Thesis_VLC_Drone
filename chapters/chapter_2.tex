The first cue\todo{i dont like this opening sentence, it feels poorly linked.} is a bearing angle from the drone to the light source. It is produced by the bearing estimation algorithm of Huang et al.~\cite{huang2026lta}, which weights the direction each photodiode faces by the light signal $S_n$ that sensor measures and sums these into a single vector; the bearing is the angle of that vector, and the full derivation is given in Section~\ref{sec:bearing}. Because the sum uses only the current sample, the bearing is available immediately, without the drone first exploring the environment.

A bearing is not always meaningful\todo{We needed this information earlier! As in before the system overview.}. When no beacon is in view, every photodiode measures only ambient light, the weighted vector has near-zero magnitude, and its angle is then set by noise rather than by the source. The magnitude of the vector therefore serves as a confidence measure for the bearing. The controller acts on a bearing only when this magnitude exceeds a minimum threshold, which prevents it from following a direction that is really noise from a weak, distant, or absent beacon.

A bearing that passes this test is still noisy between samples. Reflections, sensor noise, and small changes in attitude make the raw bearing jitter from one frame to the next, and acting on each raw value would make the heading jerk. The controller therefore smooths the bearing with an exponential moving average before passing it to the fusion stage and the state machine. The update is
\begin{equation}
\hat{\psi}_B \leftarrow \hat{\psi}_B + (1-\beta)\,\Delta,
\label{eq:bearing-filter}
\end{equation}
where $\Delta$ is the wrapped angular difference between the new bearing and the current estimate $\hat{\psi}_B$, and $\beta$ sets how much of the past estimate is retained. A larger $\beta$ rejects more noise but responds more slowly to a genuine change in direction.

Each cell stores the strongest light signal $S_n$ measured across the photodiode array at that location, using the per-sensor signal of Section~\ref{sec:lightsignal}. When the drone revisits a cell, the new measurement does not overwrite the stored value but updates it through an exponential moving average,
\begin{equation}

\end{equation}
which averages out the sample-to-sample fluctuation seen when the same location is measured repeatedly. The factor $\alpha$ weights the stored value against the new one and is set by default to weight them equally. A smaller $\alpha$ tracks new measurements more closely at the cost of more noise, while a larger $\alpha$ gives a smoother but slower-adapting map.

\section{Gradient Estimation}

The gradient gives the drone a direction to follow when the bearing is unavailable. It differs from the bearing in what it describes. The bearing points along the current line of sight to the beacon, whereas the gradient points in the direction in which the stored light field increases fastest, which stays defined even when the beacon is hidden. Huang et al.~\cite{huang2026lta} estimated this direction from a single photodiode, reading it from how that sensor's signal changed as the drone moved. That approach is slow, because the drone must keep moving to collect intensity data, and it is fragile, because a single fixed sensor measures the source only while it faces it. Estimating the gradient from all eight photodiodes removes both problems. At least one sensor faces the source whatever the drone's heading, and each location is measured directly rather than inferred from motion.

The controller estimates the gradient from the map by fitting a plane to it. Around the drone's current position it takes the cells $\{(x_i, y_i, I_i)\}$ and fits $I(x,y) = a x + b y + c$ by weighted least squares, weighting each cell by the inverse square of its distance to the drone so that nearer cells count for more. The fitted coefficients give the gradient $\mathbf{g} = (a, b)$, and the controller steers along $\mathbf{g}$ whenever it navigates by the map.

A gradient estimate, like a bearing, can be unreliable, so the controller needs a measure of how far to trust it. The plane fit provides one directly. It reports how closely the cells follow the fitted plane, through a weighted coefficient of determination
\begin{equation}
R^2 = 1 - \frac{\sum_i w_i\,(I_i - \hat{I}_i)^2}{\sum_i w_i\,(I_i - \bar{I}_w)^2},
\label{eq:gradient-r2}
\end{equation}
where $\hat{I}_i$ is the plane's value at cell $i$, $\bar{I}_w$ is the weighted mean of the cell values, and $w_i$ are the inverse-distance weights. An $R^2$ near one means the cells lie close to a single plane, so the fitted direction is trustworthy. A low $R^2$ means they do not, so the gradient should not be followed.

Three conditions must hold for the controller to act on a gradient. The fit must use at least a minimum number of cells, because a plane through two or three points is set by noise as much as by the field. Its $R^2$ must exceed a threshold, which rejects fits where the cells do not lie on a consistent slope, as happens where the field is flat or where the map is still patchy. Its magnitude must be large enough to be meaningful, so the drone does not chase a near-flat field. The thresholds, with the controller's other parameters, are listed in Appendix~\ref{app:parameters}. These three conditions define the gradient reliability used by the state machine. While they fail, the drone navigates on the bearing alone. Once they hold, the controller blends the gradient into the heading, or follows it alone when the bearing is lost.

Fitting a plane is well understood, but it is not obvious which estimator performs best on the sparse, unevenly spaced cells of a real map. We therefore implement and compare five candidate methods: ordinary and weighted least-squares plane fits~\cite{bjorck1996lsq, strutz2016wls}, a robust RANSAC fit~\cite{fischler1981ransac}, a local polynomial surface fit~\cite{bjorck1996lsq}, and finite differences~\cite{leveque2007fdm}. We judge them on three criteria, in order of importance: how often each returns a usable estimate, how accurately that estimate points, and how much it costs to compute, since the estimator runs on the drone's microcontroller. Figure~\ref{fig:gradient-comparison} reports the first two on a densely sampled reference map of the simulated light field.

We define these measures over a set of $N$ cells sampled across the reference map. At each cell a method either returns a gradient estimate that passes the reliability gates of the previous section or returns nothing; with $S_m$ the cells where method $m$ succeeds, its success rate is $|S_m| / N$. For each returned estimate $\hat{\mathbf{g}}_i = (\hat{a}_i, \hat{b}_i)$ we take the angular error against the direction straight to the source,
\begin{equation}
e_i = \left| \left( \hat{\theta}_i - \theta_i^\star + 180^\circ \right) \bmod 360^\circ - 180^\circ \right|,
\label{eq:angle-error}
\end{equation}
where $\hat{\theta}_i = \operatorname{atan2}(\hat{b}_i, \hat{a}_i)$ is the estimate's heading and $\theta_i^\star = \operatorname{atan2}(y_L - y_i,\, x_L - x_i)$ points from cell $i$ to the source at $(x_L, y_L)$, and the wrap folds the unsigned difference into $[0^\circ, 180^\circ]$. We discard cells within ten centimetres of the source, where the field is too steep to give a meaningful reference, and note that this straight-to-source reference is itself only approximate behind the pillar, where the helpful direction leads around the obstacle rather than at it. We then average $e_i$ two ways: the own-coverage error over a method's own cells $S_m$, and the common-coverage error over the cells $C = \bigcap_m S_m$ that every method estimates.

Coverage is the criterion that matters most. The bearing already gives the controller an accurate heading in open space, so the gradient does not have to be precise there; its job is to be available wherever the drone is, and above all when the bearing is lost behind an obstacle or between beacons. A method that often returns nothing leaves the controller flying blind at exactly those moments. Finite differences return an estimate almost always, at 99.7\%, but from so few cells that the direction is noisy; RANSAC succeeds least often, at 71\%, discarding too many cells as outliers; and the plane and polynomial fits sit between. Weighted least squares returns a usable estimate 96\% of the time, second only to the noisy finite differences.

The raw angular error, by contrast, is a misleading way to rank the methods, because it is entangled with coverage. A method that quietly fails on the hardest cells is never charged for them, so its average error flatters it, while a high-coverage method like weighted least squares is scored everywhere, including the cells right under the elevated source where the measured field bends sharply and behind the pillar where it leads around the obstacle rather than straight at it. This shows directly in the figures: on its own coverage weighted least squares records a mean error of 29 degrees, against 5.6 degrees for the low-coverage RANSAC fit. When we instead score every method on the same common set of cells, the cells that all of them estimate, the gap nearly closes. The plane, polynomial, RANSAC and weighted least-squares fits then agree to within about three degrees, from 5.6 to 8.5 degrees of mean error, with only finite differences staying noticeably noisier at 14.6 degrees; weighted least squares itself falls from 29 to 8.5 degrees, which shows that most of its apparent inaccuracy was simply the price of estimating the difficult cells the other methods decline. The remaining differences of a few degrees are not decisive: the bearing compensates them in open space, and the steep field under the source would not even appear in flight, where the drone follows a partial map and never measures that drop.

Computational cost and robustness settle the choice. Weighted least squares solves one small fixed-size linear system whose runtime is the same on every call, which sits comfortably within the budget of the drone's microcontroller, and its inverse-distance weighting makes it robust to the changes in overall light level that occur as the drone moves nearer to or further from the source. RANSAC, the most accurate method on its own coverage, is the most expensive and the least predictable, since it repeatedly fits and scores random subsets and its running time varies from call to call, which is awkward for a real-time control loop on a small processor. Finite differences are the cheapest but the noisiest. Weighted least squares therefore offers the best combination for this platform: the highest usable coverage, a low and constant computational cost, and robustness to light-level changes, which is why the controller estimates its gradient this way.

\section{Control Output}

The controller's primary output is a single target heading, obtained by merging the bearing direction $\psi_B$ and the gradient direction $\psi_G$. Both are angles, so the controller combines them with a weighted circular mean~\cite{mardia2000directional},
\begin{equation}
\psi = \operatorname{atan2}\!\big(w_B \sin\psi_B + w_G \sin\psi_G,\; w_B \cos\psi_B + w_G \cos\psi_G\big),
\label{eq:fusion}
\end{equation}
A plain numerical average is unsuitable here because angles wrap at $360^\circ$. Averaging $359^\circ$ and $1^\circ$ gives $180^\circ$, the opposite of the correct direction, whereas the circular mean returns $0^\circ$.

The weights $w_B$ and $w_G$ depend on which cues the state machine currently trusts. When both are reliable, the controller sets $w_B = w_G$ and the heading is their bisector. When only one is reliable, its weight is one and the other zero, so the controller follows that cue alone. When neither is reliable, the drone holds its current heading\todo{This sentece doesnt feel right. Link back to searching state instead maybe. Since we dont just hold heading, we start circling.}. The controller also sets a forward speed. The drone flies forward only when it points at the target, so the speed is zero while it turns to align and a fixed cruising value once it is aligned and approaching. Holding the speed at zero during a large heading correction keeps the drone from drifting sideways.

The heading and speed are passed to the drone's own flight controller as setpoints, and the altitude is held at a fixed height throughout.
