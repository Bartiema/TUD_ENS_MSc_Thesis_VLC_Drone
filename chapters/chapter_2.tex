\chapter{Problem \& Algorithm Design}
\label{chp:algorithm}

This chapter designs the navigation controller at the core of this thesis. Huang et al.~\cite{huang2026lta} showed that a weight-, size-, and power-constrained micro-drone can reach a single light source in open space, using either a bearing read from a photodiode array or a gradient climbed with a single photodiode. This thesis aims to fill a gap in the work of Huang at al.: reaching several modulated beacons in succession while passing obstacles between them, using only on-board light sensing. We first show the existing methods do not solve this problem. The design question is therefore how to obtain obstacle-aware, multi-beacon navigation from light alone, within the constrained budget of the micro drone platform we chose.

To overcome these limitations, our system combines two methods from Huang at al.~\cite{huang2026lta} and extends the system with spatial memory of the light field. We follow up the bearing angle estimation approach for direct light-source guidance as mentioned in their paper. For gradient estimation, however, their previous design records light intensity using only one photodiode. This design choice soon encounters a critical limitation because a drone has many degrees of freedom: it can translate and rotate continuously during flight. As the drone moves, rapid heading changes mean that the platform is not guaranteed to record valid light measurements at every position. More specifically, when the drone rotates and the photodiode faces away from the light beacon, the sensor can lose the signal for the guidance. In our approach, we instead record light signals continuously using an eight-photodiode array arranged in a circular pattern. This allows the platform to measure light intensity regardless of the drone’s heading direction. As the drone flies, the system stores the light strength measured by the photodiode array in a map. From this map, it estimates the light gradient, which provides a direction for climbing the light field toward the beacon. The remaining sections develop the controller one component at a time: the bearing estimate, the separation of simultaneous beacons by modulation frequency, the light map, the gradient estimator and the choice among candidate estimators, the state machine that sequences a mission, and the heading and speed commands issued to the flight controller.


\section{Why Existing Methods Fall Short}
\label{sec:existing-methods-fall-short}
In the system proposed by Huang et al.~\cite{huang2026lta}, bearing-angle estimation and gradient-based estimation are first developed as single-beacon navigation primitives and evaluated in open-space source-seeking tasks.This thesis examines two conditions that are more complex and closer to real-world applications than the baseline setting. The first is obstacle occlusion, where an object stands between the drone and the beacon and can block the light signal. The second is multi-target navigation, where the drone must visit several beacons in a predefined order. The following paragraphs explain why the original system is insufficient under these conditions, and the following simulations that demonstrate the failure cases addressed in this thesis.


The bearing is an instantaneous estimate, formed from the relative magnitudes of the eight photodiode signals at a single instant as explained in \ref{sec:bearing}. Because it requires no integration over the drone's motion, it yields a heading immediately whenever the array receives the beacon's light, whether directly or through a reflection. This immediacy is also its central limitation: the bearing describes only the direction of the light arriving at that instant and retains no record of the surrounding space. When an obstacle blocks the beacon, the photodiode magnitudes fall toward the ambient level, and the magnitude-weighted direction no longer points at the source. The bearing then yields no usable heading, and it gives no indication that an obstacle is present or where the beacon now lies.

The single-photodiode gradient method, the second method of Huang et al., recovers a direction using one sensor instead of an array. Because a single sensor measures only one light value at a time, this approach is slow and fragile. It cannot sense a spatial gradient directly; instead, it infers the gradient from how the measured light value changes across successive positions. As Figure~\ref{fig:single-pd-gradient} illustrates, this forces the drone to keep moving before it can estimate a direction, while also requiring the sensor to remain oriented toward the light. The method therefore depends strongly on the drone’s heading, because one fixed photodiode receives the direct beam only when it faces the source.

A single sensor reads only one value at a time, so the drone must move to compare readings $S_1$, $S_2$, and $S_3$ along its path before it can infer which way the field rises. The reading also depends on the drone's heading: at the middle position the drone has rotated away from the source, so it reads low even though it is closer than the first, which corrupts the inferred gradient. We examine these failures in simulation before testing on hardware, for two reasons. A simulated light field carries none of the ambient light or electrical noise of a real flight, so an observed failure can be attributed to the navigation method rather than to the photodiode choice limitation. The simulator also runs the same scenario repeatedly, at low cost and with no risk of damaging the aircraft, which is a critical requirement for comparing different navigation methods.

The comparison uses the Webots environment mentioned in Section~\ref{sec:simulation}, shown in Figure~\ref{fig:sim-setup}. A single beacon sits above the floor, a pillar stands between the drone and the beacon, and the drone starts on one side, so the pillar obstructs the direct path to the beacon. Each controller flies this same scene, from the same start position to the same beacon, with identical sensing and flight settings, so the navigation method is the only variable between each runs.

The bearing-only controller steers directly along the measured light direction. Since the pillar lies slightly off the direct line to the beacon, the bearing angle direction just points past its edge, and the controller follows that direction with no indication that it leads into the obstacle. The drone therefore collides with the pillar, as Figure~\ref{fig:bearing-fail} shows.

    \label{fig:bearing-fail}

The single-photodiode gradient controller also fails to reach the beacon. As Figure~\ref{fig:gradient-fail} shows, it follows a slow and irregular path and stalls when it reaches the pillar. This failure comes from how the gradient is estimated. The drone observes a local increase in light intensity only after it has already pressed against the pillar and begun moving around it. Once the sensor detects a stronger light signal after partially passing the obstacle, the gradient estimate produces a large corrective pull away from the pillar. Although this response helps the drone avoid continuous contact with the obstacle, it also pulls the drone away from the target direction, preventing it from reaching the beacon.

The two failures are showing two trade-off between each approaches. The bearing is fast and accurate but has no memory beyond the current line of sight. The gradient tolerates a blocked line of sight but is slow and must read from accumulated measurements. A new type of controller is then inspried in this thesis. We need a new controller that maintains a short spatial memory that can remembered the recorded light intensity and calcultate the gradient when there is an obstacle to pull the drone away from it and can follow the bearing angle direction once it travels away from the obstacle. The remainder of this chapter develops this controller. The results are reported in Chapter~\ref{chp:obstacle}.

Neither baseline supports more than one beacon, so neither can perform the multi-beacon traversal this thesis targets. We do not demonstrate the multi-beacon case in simulation. As Section~\ref{sec:simulation} explains, the simulator does not ray-trace the light but reads each sensor's intensity directly, and it does not reproduce the on-off modulation. Without that modulation the beacons cannot be told apart, so we chose to test beacon separation without the simulator in chapter~\ref{chp:multibeacon}.

\section{System Overview}
\label{sec:overview}

The failures of the previous section define two requirements for the controller. First, the drone must move efficiently toward a visible beacon. Second, it must continue toward the beacon when the direct path is blocked.The controller satisfies the two requirements with two separate algorithms, each addressing one requirement while using data from the same photodiode array. The first is the bearing, which gives an immediate heading whenever the array receives the beacon's light. The second is a gradient, estimated from a light map that the drone builds as it flies; because the map records where the light was stronger, the gradient still gives a direction when the beacon is not directly visible.

Neither algorithm is always trustworthy, so the controller must decide which to use at each moment. The bearing is not reliable when its magnitude is low, which happens when the beacon is weak or blocked as explained in \ref{sec:bearing-angle}. The gradient is meaningless until the drone collect enough data over spatial distribution as described in Section~\ref{sec:gradient-estimation}. A state machine compare the validity between the two algorithm and selects the drone's behaviour: it follows the bearing estimate while its signal is reliable, adds the gradient once the map is dense enough and its estimate is having high confidence, and follows the gradient alone when the bearing is invalid.

Each control update follows the same sequence, summarised in Figure~\ref{fig:controller-overview}. The per-sensor light signals $S_n$ of Section~\ref{sec:lightsignal} are first stored into a light map. The direction toward the light is then estimated from the map through gradient estimation. The bearing angle and the gradient estimation result are further merged into a single target heading through the state machine. The state machine selects the navigation strategy and forward speed for the current beacon. It also determines when the current target has been reached and when the controller should switch to the next target.

\section{Bearing Angle}
\label{sec:bearing-angle}

Before the controller can rely on the bearing, it must convert the raw estimate into a usable heading. The bearing itself is the estimator from Huang et al.~\cite{huang2026lta}, explained in Section~\ref{sec:bearing}. However, this estimator still requires smoothing and a measure of validity. This section describes the two steps the controller adds on top of it, a validity gate and temporal smoothing.

The bearing is only reliable when the photodiode array actually receives the beacon. With no beacon in view, every photodiode sees only ambient light, the vector sum has near-zero magnitude, and its angle is determined by noise. The controller therefore computes a validity score by summing the light-intensity magnitudes measured by all photodiodes in the array, and uses the bearing estimate only when this score exceeds a minimum threshold. This rejects headings that are really noise from a weak, distant, or absent beacon.

A bearing that passes this test is still noisy between samples. Reflections, sensor noise, and small changes in attitude make the raw bearing jitter from one frame to the next, and acting purely based each raw value would make the heading of the drone jerk. The controller therefore smooths the bearing with an exponential moving average before passing to the next operation to provide a more stable flying behavior. The update is
\begin{equation}
\hat{\psi}_B \leftarrow \hat{\psi}_B + (1-\beta)\,\Delta,
\label{eq:bearing-filter}
\end{equation}
where $\Delta$ is an angular difference between the new bearing and the current estimate $\hat{\psi}_B$, where $\beta$ controls the strength of the filter. A larger $\beta$ rejects more noise but responds more slowly to a genuine change in direction.

\section{Light Intensity Map}
\label{sec:light-intensity-map}

Gradient-based navigation needs a record of how the light varies across space, which the bearing alone does not required. The controller builds this record as a light map. It divides the horizontal plane into a regular grid and stores one light value per cell, indexed by the drone's estimated position in a fixed world frame. Figure~\ref{fig:light-map} shows such a map after a survey of the area.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.72\textwidth]{figures/gradient_analysis/light_map.png}
    \caption{A light-intensity map of the simulation, viewed from above. Each grid cell holds the strongest light signal recorded while the drone occupied it, drawn here as a smooth field. The strength rises toward the beacon, marked by the star, and this is the scalar field whose gradient the controller follows. The grey disc is the pillar, and the dashed circle the clearance the drone has to keep around it.
    Since the light comes from above and the light sensors are placed on the horizontal plane there is a drop in recorded light strenght when very close or bellow the light source.}
    \label{fig:light-map}
\end{figure}

To store the measured light intensity in the map, the controller must first retrieve the drone’s position and use it to index the corresponding grid cell. However, this position estimation inevitably contains error, which the algorithm must tolerate. The drone’s onboard positioning estimation drifts slowly over time, as any odometry-based estimators does. With this drift, measurements are gradually stored in the wrong cells, smearing the recorded intensity across neighbouring cells and blurring the gradient estimated from the map.


Two design choices are thus chosen to limit this error. First, the map is reset whenever the drone switches to the next beacon. The error accumulated while approaching one beacon is then discarded with its map and does not carry into the next, so drift cannot build up across a whole mission. Second, the grid is only used to estimate the heading. We do not claim to estimate the real position of the beacon in relation to the drone. This means that even if drift in the position estimate occurs during flight, the estimated heading can still remain accurate enough for navigation.


In our system, each cell stores the strongest light signal $S_n$ measured across the photodiode array at that location, using the per-sensor signal of Section~\ref{sec:lightsignal}. When the drone revisits a cell, the new measurement does not overwrite the stored value but updates it through an exponential moving average,
\begin{equation}
I_{\text{cell}} \leftarrow \alpha\,I_{\text{cell}} + (1-\alpha)\,I_{\text{new}},
\label{eq:map-blend}
\end{equation}
which averages out the sample-to-sample fluctuation seen when the same location is measured repeatedly. The factor $\alpha$ weights the previously stored value against the new one, where we set it $\alpha = 0.5$ by default. A smaller $\alpha$ tracks new measurements more closely at the cost of more noise, while a larger $\alpha$ gives a smoother but slower-adapting map.

Across the visited area the map approximates a scalar field that rises toward the beacon, because the measured signal grows as the drone nears the source. This field is the spatial memory the bearing lacks, and the next section explaines how we estimate a direction from it.

\section{Gradient Estimation}
\label{sec:gradient-estimation}

The gradient gives the drone a direction to follow when the bearing is unavailable. It differs from the bearing in what it describes: the bearing points along the light arriving now, whereas the gradient points in the direction in which the stored light field increases fastest, which remains defined even when the beacon is hidden. Huang et al.~\cite{huang2026lta} estimated this direction from a single photodiode, by reading how that sensor's signal changed as the drone moved. That approach is firstly slow, because the drone must keep moving to gather readings. Secondly the approach is fragile, because a single fixed sensor measures the source only when the photodiode is facing the light source. Both of these problems can be seen in Section~\ref{sec:existing-methods-fall-short}. Our controller instead estimates the gradient from the light map described in Section~\ref{sec:light-intensity-map}. Because the map is built from the full photodiode array, it records a usable value at each visited location regardless of the drone's heading. This removes the dependence on orientation and motion that limits the single-photodiode method.

The controller estimates the gradient from the recorded light map by fitting a plane to it. Around the drone's current position it takes the cells $\{(x_i, y_i, I_i)\}$ and fits $I(x,y) = a x + b y + c$ by weighted least squares, weighting each cell by the inverse square of its distance to the drone so that nearer cells have greater influence on the estimate. The fitted coefficients give the gradient $\mathbf{g} = (a, b)$, and the controller steers along $\mathbf{g}$ when its controlled by the gradient estimation method.

A gradient estimate, like a bearing, can be unreliable, so the controller needs a measure of its validity. The plane fitting process already provides one solution directly. We measure how well the light intensity in each cell agrees with the fitted plane using a weighted coefficient of determination.
\begin{equation}
R^2 = 1 - \frac{\sum_i w_i\,(I_i - \hat{I}_i)^2}{\sum_i w_i\,(I_i - \bar{I}_w)^2},
\label{eq:gradient-r2}
\end{equation}
where $\hat{I}_i$ is the plane's value at cell $i$, $\bar{I}_w$ is the weighted mean of the cell values, and $w_i$ are the inverse-distance weights. An $R^2$ near one means the cells lie close to a single plane, so the fitted direction is trustworthy. A low $R^2$ means they do not, so the gradient should not be followed.

Three conditions must hold for before the controller can trust a gradient estimation result. First, the fitting process must use at least a minimum number of cells, because a plane fitted through only two or three points is usually noisy and can produce a random pointing direction. Second, its $R^2$ must exceed a threshold, which rejects the fitting results where the cells do not lie on a consistent slope, as happens where the field is flat or where the map is still patchy. Last, its magnitude must be large enough to be meaningful, so the drone does not chase a near-flat field. The thresholds, with the controller's other parameters, are listed in Appendix~\ref{app:parameters}. These three conditions define the gradient reliability used by the state machine. While they fail, the drone navigates on the bearing alone. Once they hold, the controller blends the gradient into the heading, or follows it alone when the bearing is lost.

\subsection{Selection of the Gradient Estimator}
\label{sec:gradient-estimator-comparison}

Estimating a gradient from sparse and irregularly spaced map cells is a well-studied problem, but it is not obvious which estimator is best suited to the light maps produced during flight. Different methods make different trade-offs between robustness, accuracy, computational cost, and the ability to produce an estimate when map coverage is limited.

Since the gradient estimator runs continuously on the drone's microcontroller, the chosen method must not only be accurate but also execute within the available computational budget. We therefore compare several candidate estimators before selecting the one used in our experiments. Five candidate methods were evaluated:

\begin{itemize}
    \item Ordinary least-squares plane fitting (OLS) \cite{bjorck1996lsq}.
    \item Weighted least-squares plane fitting (WLS) \cite{strutz2016wls}.
    \item RANSAC plane fitting \cite{fischler1981ransac}.
    \item Local polynomial surface fitting \cite{bjorck1996lsq}.
    \item Finite-difference estimation \cite{leveque2007fdm}.
\end{itemize}

All methods estimate the local gradient from neighbouring map cells but differ in how they treat noise, sparse sampling, and outliers.

\subsubsection{Evaluation Procedure}

The methods were evaluated using a densely sampled reference map generated from the simulated light field. This reference map includes an obstacle creating a shadow. This addition let us evalute the estimators for the task they will be used for, avoiding obstacles and guiding the drone away from shadows. At each map location a method either produced a gradient estimate that passed the reliability criteria of  Section~\ref{sec:gradient-estimation} or returned no estimate. Three criteria were considered:

\begin{enumerate}
    \item Success rate: how often a usable estimate is produced.
    \item Directional accuracy: how closely the estimated direction matches the
        direction towards the light.
    \item Computational cost: the processing required for a single estimate.
\end{enumerate}

Success rate is the most important criterion because the gradient's primary purpose is to remain available whenever the bearing is lost. An estimator that is slightly less accurate but available in most of the time is more useful to the controller than a highly accurate estimator that frequently returns no result.

Directional accuracy was measured using the angular error:
\begin{equation}
e_i = \left| \left( \hat{\theta}_i - \theta_i^\star + 180^\circ \right) \bmod 360^\circ - 180^\circ \right|,
\label{eq:angle-error}
\end{equation}
where $\hat{\theta}_i = \operatorname{atan2}(\hat{b}_i, \hat{a}_i)$ is the estimate's heading and $\theta_i^\star = \operatorname{atan2}(y_L - y_i,\, x_L - x_i)$ points from cell $i$ to the source at $(x_L, y_L)$, and the wrap folds the unsigned difference into $[0^\circ, 180^\circ]$. We discard cells to calculate the error within ten centimetres of the source, where the field is too steep to give a meaningful reference.

% It is important to note that this straight-to-source reference is itself an approximation when the light signal is blocked by the pillar. Under occlusion, the useful direction is not necessarily the geometric direction toward the light source, but the direction that helps the drone move away from the pillar and recover the light signal.


Directly comparing average angular error using the same number of data points can be misleading, because the methods estimate the gradient in different ways and therefore require different amounts of local map data before they can produce a valid estimate. Some methods return an estimate almost everywhere, whereas others withhold an estimate when the local map is sparse or poorly fitted. As a result, the average error for each method is computed only over the locations where that method produces a valid estimate, rather than over the same set of time steps for all methods.

While angular accuracy is important for drone navigation, the coverage of a gradient estimator is also critical. We therefore report an additional coverage metric, defined as the estimation success rate. For each estimator, we divide the number of map cells where it produces a valid gradient estimate by the total number of cells in the map. This reports the ratio where each estimator can provide a usable navigation direction across the whole map.

To provide a fairer comparison of angular accuracy, we also evaluate the estimators on the common set of map cells where all candidate estimators return valid results. In this analysis, a cell is included only if every estimator produces a valid gradient estimate at that location. This avoids comparing methods over different parts of the map and isolates the accuracy difference among estimators through evaluating them under the same spatial conditions.


The distinction is important. Consider two methods: one returns estimates at 95\% of locations, while another returns estimates at only 70\%. The lower-coverage method can avoid many of the difficult locations simply by refusing to estimate there. Its average error may therefore appear lower, not because its estimates are inherently better, but because it is evaluated only on easier cases. The common succeess points comparison removes this effect by ensuring that every method is scored on exactly the same locations.

This behaviour is visible in the results that we will show later. Weighted least squares appears substantially less accurate when evaluated over its own coverage because it continues to provide estimates in difficult regions, including locations close to the source and behind the obstacle. When the comparison is restricted to the common subset, where all methods are evaluated on the same points, the differences between the plane-fitting methods become much smaller. This indicates that most of the apparent accuracy gap is caused by differences in coverage rather than differences in the quality of the estimated direction itself.

\subsubsection{Selected Method}

After our evaluation, we select the weighted least squares method as our final implementation for our controller. Although several methods achieve similar directional accuracy, weighted least squares provides the best balance between the three evaluation criteria. It offers high enough coverage, low computational cost, deterministic execution time, and robustness to variations in measurement density and overall light level.

Finite differences provide slightly higher coverage but at the cost of substantially noisier estimates. RANSAC produces accurate estimates where it succeeds but has the lowest coverage and the highest computational cost due to its iterative random sampling procedure. 

Weighted least squares therefore provides the most suitable compromise for real-time deployment on the drone's microcontroller and is used throughout the experiment in this work.

\section{Multi-Beacon Traversal}

In the real-world application, when multiple targets are present, the controller must be able to distinguish different beacons. We make the beacons distinguishable by modulating each one at a distinct frequency. Frequency analysis of a single photodiode signal then produces a separate peak for each beacon, allowing the photodiode array to sense multiple beacons at the same time. The traversal remains sequential. The controller selects one beacon as the current target, flies toward it, and selects the next beacon once the target is reached.


In a real-world environment, a photodiode reports a single mixed light-intensity value at each timestamp, where signals from all beacons are combined with steady background illumination. By collecting a sequence of intensity measurements over time, the controller can apply a Fourier transform to decompose the mixed signal into its frequency components. Because each beacon is modulated at a distinct frequency, the transformed signal contains an energy peak at each beacon’s modulation frequency, and the height of that peak reflects how strongly the corresponding beacon is observed. Figure~\ref{fig:fft-mock} illustrates this process using generated data for two beacons modulated at 150 and 200,Hz. Their signals overlap in the time domain but separate into distinct peaks in the frequency domain. The Fourier transform output is then passed to a peak-extraction step, which recovers the signal strength at each target frequency, where this implementation details are described in Chapter~\ref{chp:realworld}.


\begin{figure}[h]
    \centering
    \includegraphics[width=\textwidth]{figures/fft_mock.png}
    \caption{Separating two beacons by modulation frequency. This is generated data, shown only to illustrate the principle. Two lights are switched on and off as square waves at 150 and 200\,Hz, and one photodiode sees their sum as an ADC reading between 0 and 4095. Left: a short slice of that reading in time, where the two beacons are not separable. Right: the magnitude of its Fourier transform, where each beacon appears as a peak at its own frequency. The dotted lines mark the odd harmonics that a square wave also produces.}
    \label{fig:fft-mock}
\end{figure}

Each photodiode reading in the array is passed through a Fourier transform to produce its own frequency-domain response. The peak heights at the modulation frequencies differ across the array according to each photodiode’s orientation relative to the light beacon. We then use these peak heights, i.e., the frequency magnitudes, as the light-signal inputs defined in Section~\ref{sec:bearing}. The bearing estimator processes these values and produces a bearing angle toward each beacon based on its modulation frequency.


Because real-world environments may contain noisy light sources, such as computer's displays or illuminated signs, the modulation frequencies must be chosen carefully. Switching a light fully on and off produces a square wave. Although this modulates the beacon at the desired target frequency, it also introduces odd harmonics~\cite{oppenheim1999dsp}. These odd harmonics can also be seen in Figure~\ref{fig:fft-mock}. The frequency analysis separates the fundamental target frequency from its harmonics because they fall into different frequency bins. A problem arises only when one beacon’s fundamental frequency coincides with a harmonic of another beacon. In that case, the harmonic adds energy to the wrong beacon’s bin, making the two beacons difficult to distinguish. Therefore, the target modulation frequencies must be selected to avoid harmonic collisions. For two target beacons, we choose frequencies such as 150,Hz and 200,Hz.


\section{Controller State Machine}

\label{sec:state-machine}
The bearing and gradient methods each provide a direction estimate, but they do not by themselves decide what behaviour the drone should adopt at a given moment. Two situations must be considered before the drone acts. In the first, the drone may not have a trustworthy estimate, either because it has just started searching for the target beacon and the map is still empty, or because the light is blocked by an obstacle. In the second, both estimates are available, and the controller must decide which estimate to rely on while the drone moves. We resolve both situations with a navigation state machine, shown in Figure~\ref{fig:state-machine}, which consists of five states, namely \emph{searching}, \emph{aligning}, \emph{approaching}, \emph{dwelling}, and \emph{recovering}. The following paragraphs describe each state and its purpose.


In the \emph{searching} state, the drone has no trustworthy bearing or gradient estimate, and its purpose is to search for a beacon. Because the photodiode array already covers the full 360-degree horizontal plane, rotating in place would reveal little additional information. The drone therefore flies a bounded infinite sign shape search path instead. This pattern forces the drone to explore the surrounding area, increasing the chance that it passes through regions where the beacon signal is stronger while recording light measurements into the map. The drone leaves \emph{searching} as soon as the received signal produces a bearing whose magnitude exceeds the trust threshold. During this state, the drone does not use the gradient or bearing estimates to command its motion; it follows only the predefined search path while monitoring the signal strength.


Once the drone has collected enough light information to infer the target direction, either from the bearing estimate or from the gradient estimate, it transitions into the \emph{aligning} state. In this state, the drone turns toward the target while holding position, allowing it to face the beacon before committing to forward motion. It transitions to \emph{approaching} once the drone's heading falls within the alignment tolerance of the target direction, set to $15^\circ$ in this work. Appendix~\ref{app:parameters} lists this parameter together with the other controller parameters.


In the \emph{approaching} state, the drone flies forward while continuously correcting its heading toward the target. However, odometry drift can gradually shift the drone’s heading away from the estimated target direction. If the heading error later exceeds a wider re-alignment threshold of $30^\circ$, the controller returns to the \emph{aligning} state, ensuring that the drone does not continue flying forward while severely misaligned.


The heading used during \emph{aligning} and \emph{approaching} is not always determined by the bearing alone. At the start of the mission, when the map is still sparse, the drone follows the bearing estimate because too few cells have been recorded to support a reliable gradient. Once the gradient satisfies the reliability conditions in Section~\ref{sec:gradient-estimation}, the controller blends the bearing and gradient estimates into a single target heading. In this way, the drone begins using both cues during normal approach, before the bearing is lost, making the later transition to gradient-only control smoother when the bearing estimate becomes invalid.

The drone must also detect arrival at a beacon. The measured light strength rises as the drone nears the source, so the drone is treated as arrived once this strength exceeds an arrival threshold, reach. This threshold is tuned per flight, with its value listed in Appendix~\ref{app:parameters}. On arrival the drone enters the \emph{dwelling} state, holds position for a fixed dwell time of $1.0$\,s, and then moves to the next beacon. 

The \emph{recovering} state handles the case where the bearing becomes invalid in flight, for example when an obstacle moves between the drone and the beacon. The magnitude of the bearing estimate then drops below the trust threshold, and a controller without memory would have no heading to follow. However, if the gradient estimate meets its reliabilty conditions, the drone enters \emph{recovering} and continues on the gradient direction alone. Climbing the stored light field to move out of the obstacle's shadow. It returns to \emph{approaching} as soon as the bearing is valid again, and falls back to \emph{searching} only if the gradient estimate also becomes unreliable.

\section{Control Output}

The controller's primary output is a single target heading, obtained by merging the bearing direction $\psi_B$ and the gradient direction $\psi_G$. Both are angles, so the controller combines them with a weighted circular mean~\cite{mardia2000directional},
\begin{equation}
\psi = \operatorname{atan2}\!\big(w_B \sin\psi_B + w_G \sin\psi_G,\; w_B \cos\psi_B + w_G \cos\psi_G\big),
\label{eq:fusion}
\end{equation}
A plain numerical average is unsuitable here because angles wrap at $360^\circ$. Averaging $359^\circ$ and $1^\circ$ gives $180^\circ$, the opposite of the correct direction, whereas the circular mean returns $0^\circ$.

The weights $w_B$ and $w_G$ follow the validity of each input, as decided by the state machine. When both are valid, the controller sets $w_B = w_G$, so the heading is their bisector. When only one is valid, its weight is one and the other zero, and the controller follows that input alone. When neither is valid, there is no direction to command, and the drone falls back to the \emph{searching} stage rather than holding a fixed heading.

The controller also needs to set a forward speed once it has found a reliable target direction. The drone flies forward only when it is aligned with the target, so its forward speed is set to zero while it turns during the \emph{aligning} state. Holding the speed at zero during large heading corrections prevents the drone from drifting sideways. Once the heading falls within the alignment tolerance, the controller enters the \emph{approaching} state and commands a fixed forward speed toward the target. The heading and forward speed are sent to the platform’s flight controller as setpoints, while the altitude is held at a fixed height throughout the experiment. The navigation controller therefore operates only in the horizontal plane. Because it communicates with the platform only through heading and speed commands, the same navigation logic is independent of altitude in our experiments. The same implementation runs in both the Webots simulation and the real Crazyflie platform.
