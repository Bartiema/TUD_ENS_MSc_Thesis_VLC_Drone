\chapter{Future Work}
\label{chp:futurework}

This chapter sets out the directions that could build on this work.  We first consider improvements to the sensing hardware, then discuss extensions to the navigation method itself. We close by identifying the need for a broader characterisation and robustness study, which would move the current demonstrations toward a measured account of the system’s performance.


\section{Improving the Sensing Hardware}

The operating range is the most immediate target for a hardware redesign, since it is the main limitation set out in Section~\ref{sec:limitations}. A transimpedance amplifier with adjustable or switched gain would let the front end follow the signal as the drone moves. Near a beacon, where the photodiodes now saturate, it would back off. Far from it, where the signal is weak, it would amplify. The clean band would then widen at both ends. Higher-order band-pass filters tuned to the beacon band would help further. They would pass the modulation more selectively, and reject the ambient light that otherwise reduces the sensor's usable range, the effect measured in Section~\ref{sec:ambient}.

The motor-noise measurements of Chapter~\ref{chp:obstacle}, reported in Section~\ref{sec:motor-noise}, show the CrazyFlie's motors are the main cost of flight, and that vibration and electrical interference contribute in roughly equal measure. This opens a clear hardware direction. Soft-mounting the board or the motors to damp the vibration should recover the mechanical half of the loss, while better shielding and supply filtering would address the electrical half. The same measurements explain why the blimp records a higher SNR than the CrazyFlie: as Section~\ref{sec:blimp-generalisation} shows, the blimp runs its motors far less. A controlled side-by-side flight of the two carrying the same board would then measure how much of that gap soft-mounting and shielding can close.  

Photodiode geometry is another direction for future work. The current horizontal ring, described in Chapter~\ref{chp:hardware} and shown in Section~\ref{sec:pcb-layout}, was deliberately chosen to reduce the amount of overhead ambient light entering the sensors. This design, however, responds weakly to light arriving from above or below the ring plane. A geometry that distributes photodiodes over a more spherical arrangement would allow the system to sense those directions as well, although it would also admit more ambient light. A more capable sensing circuit could help manage this trade-off. The same redesign would also support faster, more agile flight and three-dimensional navigation. During aggressive manoeuvres, the drone tilts and changes the direction in which each photodiode points. In this thesis, we keep the flight gentle and near-level so that the bearing estimate in Section~\ref{sec:bearing}, which is defined in the horizontal plane, remains valid. Future work could compensate for the measured tilt and correct the bearing error introduced by more aggressive motion.


\section{Extending the Navigation}

Lifting the navigation into three dimensions would allow the drone to seek beacons above or below its current flight height, rather than only in the horizontal plane. The algorithmic extension from 2D to 3D is relatively direct. The current light map in Section~\ref{sec:light-intensity-map} stores intensity over a two-dimensional grid indexed by horizontal position. In a 3D version, this map would instead store intensity in a volume indexed by horizontal position and altitude, and the gradient estimator would include a vertical component in addition to the horizontal direction. The main challenge is that this algorithmic extension depends on sensing light variation along the elevation axis, which the current horizontal photodiode ring cannot provide. For this reason, 3D navigation should be developed together with the photodiode-board redesign discussed in the previous section, since elevation sensing cannot be achieved through software changes alone.

Future work should test the controller in more diverse obstacle settings than the single box used in Chapter~\ref{chp:obstacle}. Real deployments may contain pillars, tables, shelves, walls, or other objects that block and reshape the light field in different ways. Multiple obstacles, narrow corridors, and concave layouts could each stress the map-based recovery mechanism of Section~\ref{sec:state-machine} by creating shadows, detours, or local maxima that the stored gradient may not escape. Evaluating these cases would clarify when the current recovery strategy is sufficient and when additional planning or obstacle-awareness system is needed.


A further step would be to relax the static-environment assumption noted in Section~\ref{sec:limitations}. A beacon or an obstacle that can be dynacmic after the drone has mapped it would leave stale values in the light map of Section~\ref{sec:light-intensity-map}. Handling this would require the controller to detect that a stored cell no longer matches the field and to refresh or discard it, rather than trusting every reading indefinitely. Another direction for future work is to refine how the controller fuses the bearing and gradient estimates. The current controller uses fixed, equal weights whenever both estimates are valid, as defined by the fusion rule in Equation~\ref{eq:fusion}. When only one estimate remains valid, the state machine mentioned in Section~\ref{sec:state-machine} switches to only using that estimate. Our design is simple and straghtforward, but it treats each input as either trusted or ignored, even though both estimates already carry information about their reliability. A more flexible approach could be the controller dynamically adjusting the bearing and gradient weights, $w_B$ and $w_G$, according to their confidence. The bearing confidence could come from the summed photodiode magnitude used as the validity score in Section~\ref{sec:bearing-angle}, while the gradient confidence could come from the $R^2$ value of the plane fit in Equation~\ref{eq:gradient-r2}. With this design, the drone would rely more on the bearing when the beacon is clearly visible and more on the gradient when the bearing weakens inside an obstacle shadow, while still allowing both estimates to contribute to the final heading. Such confidence-weighted fusion would make the transition between the two estimators smoother than the current hard-threshold hand-off.

\section{Characterisation and Robustness}

The controller contains several tunable parameters, listed in Appendix~\ref{app:parameters}. Their values are currently selected empirically for the experiments in this thesis, so future work should quantify how sensitive navigation performance is to each setting. A formal ablation study would address this by repeating the same missions under fixed conditions while varying one parameter at a time and keeping the others unchanged. This study would also provide a controlled evaluation identified as missing component mentioned in Section~\ref{sec:limitations}. By measuring success rate, path length, travel time, and recovery behaviour under each parameter setting, such an ablation study would replace the current hand-tuned values with practical tuning guidelines for deploying the system in new environments.



The arrival conditions deserve a closer look before any real deployment. The controller registers arrival from a single SNR threshold, as the state machine of Section~\ref{sec:state-machine} defines. A longer mission in Chapter~\ref{chp:multibeacon}, the three-waypoint flight of Section~\ref{sec:multibeacon-traverse}, shows this can fail to latch. A more robust scheme would make the transition between beacons more dependable. It could combine the signal level with the drone's estimated proximity, or require the condition to hold over several frames before the drone commits to it.

The arrival condition deserves closer attention before real deployment. In the current state machine, described in Section~\ref{sec:state-machine}, the controller registers arrival using a single SNR threshold. The three-waypoint flight in Section~\ref{sec:multibeacon-traverse} shows the weakness of this design: although the SNR rises above the threshold near the final beacon, the state machine does not reliably latch the arrival event, and the drone continues flying. A more robust arrival mechanism should therefore avoid relying on an instantaneous SNR value alone. For example, the controller could require the SNR threshold to hold for several consecutive frames before confirming arrival. This would make beacon handover more reliable in longer missions.
